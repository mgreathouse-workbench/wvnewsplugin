## 2. Shipped this session

### 2.1 Adboard "disappearing orders" — FIXED (platform PR #17, merged + deployed)
Two bugs that made the City of Buckhannon ad (order 7911) vanish after customer approval:
- **`orders.updatedAt` type mismatch** — CRM writes ISO strings, platform wrote Firestore Timestamps. Firestore sorts Timestamps below all strings, so any board-touched order fell past the `limit(300)` and vanished. Fix: platform now writes ISO strings everywhere it touches `orders`; backfilled the 15 stranded docs (collection now 100% string-typed).
- **State machine decoupled from actions** — uploading a proof / approving didn't advance state, so a customer approval on a still-`queued` order was rejected. Fix: `addProof` auto-advances to `proof-out`; added the `queued → proof-out` edge.
- Order 7911 data-corrected to `approved` (customer had approved). It's back on the board.
- **Invariant recorded:** `orders.updatedAt` must be an ISO string, never a Timestamp.

### 2.2 Classified test data — CLEANED
Purged all 240 `ZZTEST` listings from `print_classifieds_queue`.

### 2.3 Color-placement module — BUILT + COMMITTED (plugin), not yet wired
`plugin/wvnews-print/color-placement.js` encodes the press "Color Placement Configuration": given format (broadsheet|tab), total page count, and folio, `isColorPage()` returns whether the page is Full Color. Broadsheet folios are section-letter-first (`B3`), page numbers reset per section. Unknown configs return `null` (caller warns). Behavior when a page is BW: **show the color ad but warn** (a color→BW conversion tool comes later). Waits on ad data reaching the plugin.

## 4. Reference findings (System behavior)

### 4.1 Publications
27 publications in the System, each with a `format` field (`broadsheet` | `tabloid` | `magazine` | `digital` | `pamphlet`). The plugin can read `format` straight off the publication record for color-table selection — no separate tab/broadsheet list needed. ~19 broadsheet, a few tabloid, rest magazine/digital/pamphlet.

### 4.2 Order `type` (Print vs Package)
Order #1 is `Print` (legacy migrated Newzware ad, ~6,900 of them). Every new order came out `Package` because the **Proposal Builder** (`wvnews-crm/src/components/ProposalBuilder.jsx`) derives `type` from the catalog product's category via `typeForProduct()` — `display → Package` (display products bundle print + digital). There's no Type dropdown in that flow. **Brian has since fixed this** (`fix(orders): order type shows selected product instead of always 'Package'` + `scripts/backfill-order-type.mjs`) — leaving it be.
- **Takeaway for the plugin:** filter display ads by `catalog.category === 'display'` or a populated `print.*` block, NOT by `type`.

### 4.3 How an ad reaches Published
Only a human clicking "→ Published" in the adboard drawer. The `publish-scheduled` cron only publishes *stories*, never *orders* — the "cron flips when run date passes" comment for ads is aspirational. This is what §3.1 aims to replace.

## 5. Open / blocking work (the "major changes elsewhere" first)

1. **Orders have no per-edition run date** — retail display orders are authored in the CRM, assigned publication/page/date later in `/admin/autopage`, not at creation. For the plugin to pull "Scheduled ads for this edition," a scheduled order must carry publication + edition/run-date. Fast v1 fallback: pull all Scheduled ads by **publication**.
2. **Adboard state machine vs. autopage are decoupled** — two parallel systems assigning orders to pages; need a single source of truth for "this ad belongs on this page of this edition."
3. **`approved → scheduled` is an empty state flip** — must capture the target edition.
4. **Three clashing paper-ID schemes** — numeric `print.paper`, slug ids, legacy ids; must canonicalize (`canonical-pub.js` is a start) before "ads for this publication" is reliable.
5. **Print-ready source guarantee** — confirm designers always upload a placeable print-ready source before an ad is schedulable.
6. **Adboard `listOrders` limit:300** (DEFERRED) — replace with a state-scoped query at scale.
